| `mysearch/clients.py` | 搜索编排与 provider 适配层 | 统一处理 intent、strategy、routing、cache、blended search、extract fallback、research workflow。当前运行时已经显式固化 `mode -> primary -> provider 介入，不再是默认混排常驻项。结果层同时拆开了 `web` 与 `news` 排序：`news` 更看发布时间、媒体质量和新闻页形态，`web` 更看官方性与页面相关性。搜索结果层继续补 `source_diversity / official_source_count / confidence / conflicts` evidence；`include_domains`、明确 `official/官方/官网`、以及 `docs/pricing/changelog` 这类资源语义统一进入 `official_mode=strict`，命中官方候选时会收束到官方项，未命中则保留结果并显式打出 `strict-official-unmet`。strict official 结果现在还会在 Exa rescue、Tavily PDF boost 与 PDF title enrichment 之后再次进入统一 finalize 阶段，避免后续 boost 把官方页重新冲掉；同时 `OpenAI API Pricing` 会显式压低 `ChatGPT Pricing / Business ChatGPT Pricing` 这类错误 canonical 页。对于少数高价值 troubleshooting docs，如果 strict 模式下 provider 结果里仍没有官方候选，运行时会注入小范围 canonical rescue，例如 `Playwright strict mode violation fix -> https://playwright.dev/docs/locators`、`Next.js hydration error -> https://nextjs.org/docs/messages/react-hydration-error`；而 `status` 类严格官方题如果 provider 只剩非 status 官方页或直接空结果，也会注入 brand status root，例如 `https://status.openai.com/` 或 `https://www.cloudflarestatus.com/`，并在 evidence 里留下 `official_rescue_applied / official_rescue_source=canonical-map`。当这类 docs/github/pdf/resource/tutorial 查询没有 `answer` 时，summary fallback 现在不再只停留在 `Top official match` 一行，而会把首条结果的 snippet/excerpt 一并带出来，避免多语言官方文档或 API 参考页看起来“命中对但信息太薄”。`news / entertainment / status` 里的 award / result-event query 还会在 provider summary 被泛娱乐报道带偏。`research` 返回的 evidence 现在会继续补 `research_plan`、`search_confidence`、`page_success_rate`、`page_error_count`、`exa_discovery_count`、`exa_unique_url_count`、`exa_promoted_page_count`、`selected_candidate_cluster_counts`、`source_cluster_count`、`cross_provider_candidate_count`、`provider_match_depth` `https://www.firecrawl.dev/compare` compare hub，并把 `Apify API` 这类对手方 first-party docs 一起带进 supporting 候选，不再只剩第三方目录页；而且这些 known pair / compare hub 结果现在会在注入 `canonical_research_docs` 源头时就排到 vendor docs 前面，同时在 fallback 最终化阶段再做一次 project-first 重排，避免后续 strict/resource rerank 再把 first-party comparison page 压回 supporting docs 后面。non-authoritative comparison research 里的 supporting docs `Summary:`、`TL;DR:`、JS code shell、CTA 和导航壳，再优先取第一条真正有比较信息的句子；当同一来源被多 provider 同时命中时，claim 也会升级到 `cross-provider`。Firecrawl-primary 的 blended primary 现在在 `verify/deep` 也会带正文，不再只有 `deep` 才做正文级验证。对于 X/Social，official 模式先读 `https://status.x.ai/` 的状态页；只有状态获取失败或无法判定时，才会回退到一次固定 `grok-4.1-fast` 的轻量 `/responses` 探针。compatible 模式则把当前配置的上游地址统一裁到根路径后探 `根目录 + /health`；只有根 `/health` 无法使用时，才会回退到一次显式指定 | `proxy/server.py` | Proxy 与控制台入口 | 暴露 Tavily / Firecrawl / Exa / Social 代理端点、控制台页面与管理 API；除了默认 `/` 搜索控制台以外，现在还额外提供 `/mysearch` 作为独立的 MySearch 接入页。FastAPI 入口已经切到 lifespan：启动时初始化数据库，退出时统一关闭共享 `http_client`，不再依赖已废弃的 `startup` hook。导入期的 `asyncio.Lock()` 也改成惰性初始化，避免 Python 3.9 在测试或脚本导入阶段留下未关闭事件循环。控制台模板渲染当前使用的是新式 `TemplateResponse(request, template_name, context)` 签名；如果仍按旧顺序把 `template_name` 放在第一个参数，首页 `/` 和 `/mysearch` 会在 Jinja cache key 阶段直接报 `TypeError: unhashable type: 'dict'`。Tavily 上游链路现在会先按当前配置发起 probe 和真实转发；如果上游模式下首次命中 `404`，还会自动回退到 `/api/tavily/*` 兼容路径，避免 Hikari 类网关在 `Base URL` 写成主机根时直接报 `Not Found`。当 Tavily 当前实际走上游时，控制台会先读公开的 `/api/summary`；如果额外配置了 Hikari admin headers 或 admin cookie，再继续读 `/api/keys` 聚合上游 key / quota 细项。没有 admin 认证时，界面会明确停在“public summary only”，不再假装 per-key 细项可用。Social / X 则继续分成两档：接通后台 admin 接口时返回完整 token 统计；只有手动上游 key / gateway token 时，只返回 `upstream_visibility` 这类基础接线可视化，不再把“没有后台 token 面板”和“没有接通上游”混成同一组 0 值。grok2api 后台路径现在默认优先探测新版 `/admin/api/config` 与 `/admin/api/tokens`；只有新版不存在时，才自动回退到旧的 `/v1/admin/*` 和 `/api/v1/admin/*`。token 接口如果不是直接返回 `pool -> list`，而是包了一层 `{tokens:{...}}`、`{data:{...}}`、`{items:[...]}` 之类 envelope，当前解析逻辑也会先解包再做统计，避免后台已接通但控制台仍显示一排 0。Social/X 的 fallback 现在也不会再因为 `fallback_min_results=3` 和 `max_results=1` 这种组合而必然触发第二次模型调用；实际阈值会按 `min(max_results, fallback_min_results)` 收束。现在还额外提供了受限的 `/api/internal/mysearch/token` bootstrap 接口：仅在配置 `MYSEARCH_PROXY_BOOTSTRAP_TOKEN` 时启用，用于给 `mysearch` 容器自动创建或复用专用的 `mysp-` token，不让运行时直接读 SQLite。来源：proxy/server.py, mysearch/social_gateway.py |

## 关键配置族

- `MYSEARCH_PROXY_*`
  - 统一下游接入 `proxy-first` 的最小配置；`MYSEARCH_PROXY_BASE_URL` 和 `MYSEARCH_PROXY_API_KEY` 会被 `MySearchConfig.from_env()` 读入，并继续影响 Firecrawl / Exa 的默认 base URL 与 path。对 Tavily 来说，它们只会落到 `gateway` 语义，不再覆盖显式 `official` 模式。来源：README.md:178, mysearch/config.py:281, mysearch/config.py:287, mysearch/config.py:320, mysearch/config.py:421
- `MYSEARCH_MCP_*`
  - 控制 MySearch 自身的 transport、host、port、mount path、SSE path、streamable HTTP path。来源：install.sh:21, mysearch/__main__.py:12, mysearch/config.py:253
- `MYSEARCH_TAVILY_MODE`、`MYSEARCH_TAVILY_*`、`MYSEARCH_TAVILY_GATEWAY_*`
  - Tavily 的显式双分支配置。`MYSEARCH_TAVILY_MODE=official` 时走官方 Tavily key、本地 key 列表或 key 文件；`MYSEARCH_TAVILY_MODE=gateway` 时走上游 gateway token、独立 base URL/path/auth 配置，适合 `tavily-hikari` 这类上游。来源：install.sh:27, install.sh:38, mysearch/config.py:198, mysearch/config.py:220, mysearch/config.py:320, mysearch/config.py:382
- `MYSEARCH_FIRECRAWL_*` / `MYSEARCH_EXA_*` / `MYSEARCH_XAI_*`
  - Firecrawl / Exa / xAI 的直连或兼容网关配置；xAI 继续用 `official` / `compatible` 区分官方与 social gateway。其中 `MYSEARCH_XAI_MODEL` 主要影响 official `/responses` 的正常搜索；compatible `/social/search` 的正常搜索默认仍看网关侧 `social model / fallback model`。来源：install.sh:47, install.sh:58, install.sh:68, mysearch/config.py:423, mysearch/config.py:462, mysearch/config.py:501

## X / Social 模型来源速查

- official 正常搜索：看 `MYSEARCH_XAI_MODEL`
- compatible 正常搜索：默认看 Proxy / social gateway 控制台里的 `social model` 与 `fallback model`
- official health probe：先看 `status.x.ai`；只有状态获取失败或无法判定时，才固定回退到 `grok-4.1-fast`
- compatible health probe：先看根 `/health`；只有根健康端点不可用时，才固定回退到 `grok-4.1-fast`

这里要特别区分“正常搜索模型”和“health fallback 模型”：

- 控制台里的 `model / fallback model` 控制 compatible `/social/search` 的正常搜索与补搜
- `MYSEARCH_XAI_MODEL` 控制 official `/responses` 的正常搜索
- health probe 的 `grok-4.1-fast` 只是轻量探针，不会覆盖正常搜索时的模型选择
- Proxy API
  - Tavily 走 `/api/search` 和 `/api/extract`；Firecrawl 走 `/firecrawl/v2/search` 和 `/firecrawl/v2/scrape`；Exa 走 `/exa/search`；Social 走 `/health`、`/social/health` 和 `/social/search`。其中根 `/health` 现在是 `proxy-first` 下 X compatible 健康探针的统一入口，便于把 `http://host:8000/v1`、`https://host/admin` 这类地址都裁回根目录后统一探测。来源：proxy/README.md:31, proxy/README.md:45, proxy/README.md:59, proxy/README.md:90, proxy/server.py:47
- Settings Test API
  - 控制台侧现在还额外暴露 `/api/settings/test/tavily` 与 `/api/settings/test/social`，给设置中心的“测试当前连接”动作使用；前者会按 Tavily 当前模式解析结果执行 live probe，后者会返回 Social / X 上游的 token 来源、后台连通性与可用凭证诊断。两条接口现在都会补 `request_target`、`auth_source`、`status_label`、`failure_reason`、`recommendation` 这类结构化字段，直接供前端渲染 probe 卡。来源：proxy/server.py:429, proxy/server.py:2259, proxy/server.py:2267
