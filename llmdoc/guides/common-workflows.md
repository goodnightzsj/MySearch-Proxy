## 2. 部署 proxy-first 链路

1. 现在最省事的部署方式是单容器镜像 `mysearch-stack`：直接暴露 `9874` 和 `8000` 两个端口，容器内部同时启动 `proxy` 与 `mysearch`，`proxy` 默认对外监听 `9874`，`mysearch` 再通过容器内回环地址访问 Proxy。来源：README.md:194, Dockerfile.stack:1, docker/combined-entrypoint.sh:1
2. 如果你更看重服务边界清晰，仓库根目录的 `docker-compose.yml` 仍然提供“两服务一套 stack”：`proxy` 与 `mysearch` 在同一个 compose 网络里启动，`mysearch` 默认通过 `MYSEARCH_PROXY_BASE_URL=http://proxy:9874` 回连同 stack 内的 Proxy。来源：docker-compose.yml:1
3. 不管是单容器还是 compose，两种部署现在都不再要求你手动先创建 `mysp-` token。`proxy` 只要配置了 `MYSEARCH_PROXY_BOOTSTRAP_TOKEN`，就会启用受限的 `/api/internal/mysearch/token`，`mysearch` 启动脚本会自动用同一个 bootstrap token 去创建或复用自己的专用 `mysp-` token。初始化时你只需要登录控制台补 provider 配置和 usage sync。来源：proxy/server.py:680, proxy/server.py:2702, mysearch/docker-entrypoint.sh:1, mysearch/scripts/bootstrap_proxy_token.py:1

## 4. 排查搜索行为

1. 先跑 `mysearch_health`，确认 provider 是否可用，而不是只看 key 有没有填。来源：mysearch/server.py:156, README.md:105
2. 再看 `route` 与 `route_debug`，判断是显式 provider、生效的 intent/strategy，还是 blended/hybrid 路由。当前 docs/resource 默认不再自动 blended；如果你看见 `route_debug.domain_filter_mode=site_query_retry`，说明 Tavily 的官方域约束已经进入了 `site:` 形式重试，而且第二轮已经改成“放宽 provider 侧过滤、再在客户端做域名筛选”，不再把同一层 `include_domains` 直接重放给上游。`route_debug` 里现在还会带 `requested_max_results` 和 `candidate_max_results`，表示 `strategy` 是否扩大了候选池预算。来源：mysearch/clients.py:359, mysearch/clients.py:651, mysearch/clients.py:1013, mysearch/clients.py:1484
3. 文档类查询要分清“页面发现”和“正文抓取”是两个阶段；正文异常先查 Firecrawl，再看 Tavily/Exa fallback。纯 `pricing/价格` 这类官方价格题当前不再默认按 docs/resource 路由，除非 query 里还显式带了 `docs`、`documentation`、`manual` 之类文档信号。对于 `linux.do`、`zhihu.com`、`medium.com` 这类受限 / 社区域名，如果又叠加 docs/resource 语义，运行时现在会优先把页面发现切到 Firecrawl；但精确 official/docs/pricing/changelog 这类 canonical 页查询会继续保持 `Tavily 先发现 -> Firecrawl 再验证/抽取`，即使调用方已经要求 `include_content=true`。来源：mysearch/clients.py:1023, mysearch/clients.py:1101, mysearch/clients.py:677, mysearch/clients.py:3489
4. 如果结果本身看起来“搜到了，但不够可信”，直接看 `evidence`。当前搜索结果已经统一补了 `source_diversity`、`source_domains`、`official_source_count`、`official_mode`、`confidence` 和 `conflicts`；其中 `official_mode=strict` 表示这次查询要么显式带了 `include_domains`，要么 query 明确要求“官方/官网/official”，或落在 `docs/pricing/changelog` 这类严格资源语义。strict 模式下如果已经命中官方候选，结果会自动收束到官方项；如果仍然没有官方命中，则不会再把结果过滤成空，而是通过 `strict-official-unmet` 明确告诉你“这次官方约束没有满足”。`web` 与 `news` 现在也会分开排序：`news` 更重时效与媒体质量，`web` 更重官方性与页面相关性。来源：mysearch/clients.py
5. 如果 docs / github / pdf / tutorial 的结果“命中对了，但解释看起来太薄”，先看 summary fallback 是否已经带出 snippet/excerpt。当前这类查询在没有 `answer` 时，不再只显示 `Top official match` 或 `Top source` 一行，而会把首条 snippet/excerpt 拼进去；如果仍然很薄，通常说明 provider 本身只返回了命中标题而没有正文片段，而不是排序错了。来源：mysearch/clients.py
6. `research` 现在要额外看 research 级 evidence，而不只是 `pages` 列表。返回里已经补了 `research_plan`、`search_confidence`、`page_success_rate`、`page_error_count`、`official_mode` 和 research 级 `confidence/conflicts`：如果搜索本身很强但正文抓取不完整，会出现 `page-extraction-partial`；如果是严格官方查询却没命中官方候选，会直接暴露 `strict-official-unmet`。同时 `research` 会按 query 类型自动调 `web_max_results / social_max_results / scrape_top_n`，docs 题更保守，新闻题和 exploratory 题更发散。`deep` 策略下 Exa 并行发现现在也会正式记入 evidence：`exa_discovery_count` 表示 Exa 发现条数，`exa_unique_url_count` 表示 Exa 的独立 URL 数，`exa_promoted_page_count` 表示最终被提升进正文抓取队列的 Exa 页面数。research evidence 还会把 `authoritative_source_count` 与 `supporting_source_count` 分开：supporting 来源不会再把报告写成“authoritative sources”，而 comparison claim 也会优先用有内容的 excerpt，而不是目录页标题、CTA、授权页壳或 markdown boilerplate。authoritative-preferred docs research 里，generic query 的 `agentic/search/retrieval` 这类泛词不再当品牌词，marketing/blog 页会被降到 `general/community`；当 shortlist 已经有 official/supporting 锚点时，general 候选也会限流，避免 report 前排重新被 vendor/blog 塞满；generic vendor-doc query 下，`official/supporting` 候选还会按注册域名限流，避免同一 vendor 的多条 docs 挤掉其他 first-party docs。non-authoritative comparison research 里，只有 direct first-party comparison 页会升为 `project`，同域 `alternatives/pricing` 这类 branded marketing 页会降到 `listicle`，同域 `project` 结果也会限流，避免 shortlist 被单一 vendor 刷满。debug 时还要区分 `search_authoritative_source_count`（搜索结果层命中的官方候选）与 `selected_authoritative_source_count`（真正进入 shortlist 的 authoritative 候选），不要再把“搜索里碰到官方页”和“报告实际用了 authoritative 锚点”混成一个值。对 generic vendor-doc query，如果 primary web discovery 和 docs rescue 都失败，现在会直接退到 `canonical_research_docs`，至少返回 `Tavily / Exa / Firecrawl` 的 first-party docs 证据束，而不是整条 research 直接抛错；如果 query 本身就是 vendor comparison，这个 canonical fallback 还会补一条已知的 first-party comparison page，例如 `Firecrawl vs Tavily`。来源：mysearch/clients.py
7. `social/x` 现在除了独立 `120s` 超时外，还会对兼容 gateway 的瞬时 `502 / 503 / 504 / TLS connect error / timeout / EOF` 最多重试 3 次；如果重试后仍失败，会先查同 query 的 `social_last_good_cache`，只有 cache 也没有时才掉 `tavily_social_fallback`。fallback 结果会限制同一 handle 的刷屏，citation 也只保留与当前结果对齐的 x.com 链接；而且如果完全相同的 social query 刚刚已经通过 `tavily_social_fallback` 拿到有效结果，后续重复查询会直接命中这条 social cache，不再把整条 `xAI -> fallback` 退化链重跑一遍。如果你在 debug social 题，先看 raw 里的 `fallback.to`：`social_last_good_cache` 说明 xAI 主链最近曾经成功、这次只是瞬时错误；`tavily_social_fallback` 才说明当前连 cache 都兜不住。纯 `mode=social` 的 evidence 现在还会补 `social_identity_diversity`、`social_handles` 和 `diversity_basis`；如果 `source_diversity=1` 但 `social_identity_diversity>1`，说明这是单域多账号的真实社媒流，不该再按普通 web 的低来源多样性去判。来源：mysearch/clients.py
7. `verify/deep` 下如果 evidence 已经暴露 conflicts、且至少有两方 provider 参与，而当前查询又不是 strict official，runtime 现在会让 xAI 进入冲突仲裁层：它不会直接改写结果列表，而是给出 `xai_arbitration_summary / xai_arbitration_confidence`，必要时才把这段仲裁摘要挂到 `answer`。因此排查“为什么这次 verify 看起来更像解释冲突而不是继续搜更多结果”时，先看 evidence 里是否出现 `arbitration_source=xai`。来源：mysearch/clients.py
8. 如果问题出在团队共享链路，而不是本地 runtime，就把排查重心切到 `proxy/` 的 key 池、token、usage sync 和 social gateway 配置。Social / X 现在推荐只填 grok2api 后台根地址，控制台与 `mysearch.social_gateway` 会优先探测新版 `admin/api/config`、`admin/api/tokens`，只有新版不存在时才自动回退到旧的 `/v1/admin/config`、`/v1/admin/tokens` 与 `/api/v1/admin/*`；因此如果测试结果里出现 `/v1/admin/config -> Not Found`，优先说明进程还没重启到新代码，而不是操作者一定填错了地址。token 状态统计仍兼容 `{tokens:{...}}`、`{data:{...}}`、`{items:[...]}` 这类 envelope，所以“后台地址和 app key 都已配置，但 Social / X token 统计还是 0”通常表示返回结构已经偏离 admin token 语义，或当前实例其实不是 grok2api 后台。Tavily upstream 这边也要分两档：只填 `tavily-hikari` 上游 Base URL 和 access token 时，控制台只能稳定读取 public `/api/summary`；如果你还想让控制台继续显示上游 key 列表和 quota 聚合，就必须额外补 Hikari admin headers 或 admin cookie。另一个常见误判是“设置里的 Social/X 测试连接正常，但 `mysearch_health` 里 xai 超时”：设置测试只证明后台管理面和上游 key 解析正常，不等于真实搜索路径一定能在 10 秒内跑完。当前 runtime 已经把 X probe 分成两档：official 模式先读 `status.x.ai`，只有状态获取失败或无法判定时，才回退到一次固定 `grok-4.1-fast` 的轻量 `/responses`；compatible 模式把当前上游地址裁到根目录后统一探 `/health`，只有根 `/health` 不存在或不可用时，才回退到一次固定 `grok-4.1-fast` 的轻量 `/social/search` 探针。这里的 `grok-4.1-fast` 只用于健康探针；compatible 正常搜索仍默认看控制台里的 `social model / fallback model`，official 正常搜索仍看 `MYSEARCH_XAI_MODEL`。另外，`MYSEARCH_XAI_SOCIAL_TIMEOUT_SECONDS=120` 现在表示整条 social 查询的总预算，而不是让 `xAI` 单独吃掉完整预算：如果 `xAI` primary 已经超时，runtime 会更早把余量留给 `tavily_social_fallback`，并且 fallback 本身会对一次瞬时 `502/proxy_error` 再试一次。所以现在看到 `xai request timeout after 45s` 不代表配置没生效，而是说明 primary budget 已按预期被提前截断；如果同题最终仍是 `social_unavailable`，优先检查的是 `tavily_social_fallback` 是否也同时 `502`，而不是再把锅先甩给 xAI gateway。来源：proxy/server.py, mysearch/social_gateway.py, mysearch/clients.py
9. `news / entertainment` 的 winner 类查询如果 top result 已经是官方奖项页，但 snippet 和常规正文抓取还没给出 winner，runtime 现在会再对 `oscars.org / grammy.com` 这类官方奖项页做一次原始 HTML 文本抽取，把压平的 `Best Picture Winner ...` 结构补成显式 `answer`。所以像 `2026 Oscars best picture winner`，当前预期已经不该再只剩 `Top news match: The 98th Academy Awards | 2026 - Oscars.org`，而应直接补成 `Best Picture winner: One Battle after Another`。来源：mysearch/clients.py
9. 调试 `research` 报告时，要分清三层：`claim_evidence`、`claim_level_evidence`、可见 report sections。当前 `claim_level_evidence` 与 `claim_evidence` 已经对齐，都会给每条 claim 补 `support_basis`；如果 claim 还是 `single-source`，但 `support_basis=shortlisted vendor docs` 或 `shortlisted comparison page`，说明这条证据已经有清晰锚点，只是还没到跨来源 corroboration。反过来，如果同一来源被多个 provider 同时命中，claim 现在也会升级到 `cross-provider`。同时，比较页的 claim 会先去掉 `Summary:`、`TL;DR:`、JS code shell、CTA 与导航壳，再选择第一条真正有比较信息的句子；`Executive Summary` 与 `Recommendation` 不再盲目取 `claim_evidence[0]`，而会优先使用第一个“非 generic、非 single-source、且具实质句意”的 claim。对于 comparison query，summary 还会优先把 shortlist winner 的 `candidate + fit` 放在第一句，再把实质 claim 当作支撑理由接在后面，并补一句 authoritative/supporting support summary。现在还额外会显式过滤 templated JSON-shell / API schema 壳，例如 `{{ "id": ... "completion_window": ... }}`；如果 page excerpt / snippet / content 都只剩这种壳文本，claim 组装会直接丢弃，不再把它重新塞回 `Consensus Snapshot`、`Claim-Level Evidence`、`Ranked Shortlist` 或 `Decision Table`。对 `compare OpenAI Responses API and Batch API for long-running tasks 2026` 这类 canonical OpenAI comparison，当前预期是：可见 summary 先给 `Responses` / `Batch` 的适用边界，再把 `Batch API is designed for bulk asynchronous workloads...` 这类实质句子作为 supporting evidence；如果又退回 community thread 或 JSON/schema 壳，优先检查 canonical OpenAI docs 注入和 report 组装，而不是先改 provider 主路由。来源：mysearch/clients.py
10. 现在做内部双 MCP benchmark 时，Tavily comparator 已回到直接 MCP 链路：`initialize -> notifications/initialized -> tools/call`。runner 不再自动切 Tavily-compatible REST fallback，也不再把 Tavily 故障折叠成 `comparator-blocked`；如果 Tavily 自己返回错误，就按 Tavily comparator 的直接失败暴露出来。对于 `mysearch` / Tavily 两侧偶发的 `Session not found` / `Missing mcp-session-id`，runner 现在也会自动重新 `initialize` 一次再重放 `tools/call`，避免容器刚重启后把 benchmark client 自己的 session 丢失误判成 runtime 回归。来源：scripts/run_remote_mcp_benchmark.py
11. 当前恢复后的实测状态是：`tavily_search` 已能直接通过 MCP 返回结果，例如 `official-web-precision-01` 会直接命中 `https://openai.com/api/pricing/`；但 `tavily_research` 在 `research-quality-01` 上仍会直接暴露 `HTTP 502`。这类 research 上游错误现在不会再被内部 fallback 或 blocked 分类吞掉，调试时要把它视作 Tavily comparator 当前的真实状态。strict official / exact docs 的 discovery 这边也新补了一层轻量化：如果 route 已经明确是 “Tavily 先发现 canonical 页，再让 Firecrawl 做验证/抽取”，那么 dispatch 不再把请求里的 `include_content=true` 原样透传给 Tavily；真正发给 Tavily discovery 的是 `include_content=false + strategy=fast`。这条修正已经在 live runtime 上验证过：`strict-constraint-precision-01` 现在直接是 `provider=tavily / route_provider=tavily / fallback=None / Top1=background guide`，不再走 `45s timeout -> Firecrawl fallback -> canonical-map rescue`。如果这类题又出现旧链路，优先检查当前运行环境是不是还没部署到 `d562803` 之后的代码。来源：scripts/run_remote_mcp_benchmark.py, mysearch/clients.py
9. 如果容器日志里出现 `GET / HTTP/1.1 500` 且堆栈落在 `TemplateResponse` / `unhashable type: 'dict'`，优先怀疑 `proxy/server.py` 还在用旧的模板渲染参数顺序。当前 Starlette/FastAPI 需要走 `TemplateResponse(request, template_name, context)`；如果把 `template_name` 放在第一个参数，首页 `/` 与 `/mysearch` 都会直接 500。另一个容易混淆的现象是 `/mcp 404`：这通常不是 `mysearch` 挂了，而是请求打到了 `proxy` 的 `9874`。`proxy` 本来就不提供 `/mcp`，MCP 入口应当继续走 `8000/mcp` 或宿主机映射后的 `18000/mcp`。来源：proxy/server.py, mysearch/server.py

## 5. 什么时候改哪里

- 改 prompt、安装话术、AI 调用顺序：看 `skill/`
- 改 runtime 参数、provider 路由、search/extract/research 组合：看 `mysearch/`
- 调试 comparison research 时，优先看 shortlist 里的 supporting docs 是否已经拿到 canonical vendor-doc snippet；如果 `Claim-Level Evidence` 又被 `agent-skills`、`copy markdown`、`skip to main content` 这类壳页抢走，优先修 report/claim 组装，不要先去改 provider 主路由。
- 改统一 token、代理 API、控制台和持久化：看 `proxy/`
- 改 OpenClaw bundle：看 `openclaw/`

## 奖项 / 结果型事实题
- 优先：`search(query="...", mode="news", strategy="verify")`
- 当前 award / result-event query 在主结果只有弱娱乐页时，会保持 Exa rescue 开启；只有 strong winners-page signal 才会短路 rescue。

## Research 可见层噪声
- 如果 comparison research 的 `Ranked Shortlist` / `Decision Table` 又出现 `* [Models]...`、`# Batch API | OpenAI API [![Image...` 这类导航壳或 badge-shell，优先检查 comparison row note 的 fallback 是否还停在 excerpt 层。当前 runtime 已把这类 markdown link-index 壳视为可见层噪声；对 `official` row，会优先回落到 claim text 或 cleaned title，所以报告现在预期应该更像 `Migrate to the Responses API / Batch API / Create batch` 这种短证据，而不是导航碎片。来源：mysearch/clients.py
- strict official / exact docs 的普通 `summary` 也要按同样口径看：如果 `Top official match: ...` 后面又跟了 `Guides and concepts for the OpenAI API ...`、代码片段或导航碎片，优先修 `_search_summary_excerpt` 的可见层去噪，而不是先改 provider 路由。来源：mysearch/clients.py
